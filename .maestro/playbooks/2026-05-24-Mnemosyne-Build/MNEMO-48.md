# MNEMO-48 — Messaging: Group threads — `ThreadCoordinator` DO, triage gate, floor control, loop prevention

Phase 48, closing Track H (see `MNEMO-00-ROADMAP.md`). Goal: multi-agent group threads where a message may
reach **one or more agents** (and humans), every agent **sees every message** but **responds only when it
has something valuable** — no pile-on, only signal. A `ThreadCoordinator` Durable Object (one per group
thread) fans each message to every member agent's cheap Haiku **triage gate**, collects confidence **bids**
within a debounce window, and lets only the **top 1–2** run the full loop and reply; an **@-mention forces
a response**; agent↔agent runaway is prevented by a hard cap on agent turns per human turn + a post-speak
cooldown. Depends on MNEMO-46 (`onInboundMessage`, the loop, persistence helpers incl. `getOrCreateGroupSession`)
and MNEMO-47 (access control — `group_member` tier; `expandWhitelistForGroup`). Per `docs/PRD.md` v0.5 §9.4
(triage gate → floor control → @-mention override → loop prevention) and §9.5 (group transcript keyed by
`(agentId, threadId)`, full multi-party history, each message tagged with `from` + `channel`).

Conventions: the coordinator is a new `ThreadCoordinator` DO, one per group thread via
`env.THREAD.idFromName(threadId)` (same `idFromName` idiom as the agent DO, MNEMO-04). Its SQLite uses the
`src/audit/store.ts` `SCHEMA: string[]` init pattern. The triage gate is a **cheap, separate** model call
(Haiku-class) — NOT the full agent loop — and resolves via the existing `getModel()` resolver (MNEMO-13)
with a fixed cheap model, isolated in `src/messaging/triage.ts`. The coordinator orchestrates; each agent
keeps its own DO (identity/memory/tools). @-mention parsing and loop-prevention rules are pure functions so
they are unit-testable without DOs.
