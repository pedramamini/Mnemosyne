# MNEMO-44 — Messaging: `MessagingChannel` interface + `TwilioSmsChannel`

Phase 44, opening Track H — Messaging: SMS via Twilio (see `MNEMO-00-ROADMAP.md`). This is a **paid
per-agent add-on, a later phase** — the core (memory/research/reporting/audit/brain) ships first. Goal:
the provider-agnostic seam for the messaging channel and the only concrete implementation we ship now,
`TwilioSmsChannel`, which sends outbound SMS via the Twilio Programmable Messaging REST API and handles
GSM-7 segmentation. Depends on MNEMO-15 (the agent loop produces the reply text this channel sends) and
MNEMO-01 (Worker scaffold, `src/env.ts`, vitest workers pool). Per `docs/PRD.md` v0.5 §9.1 + §9.3:
Twilio is the channel; keep the `MessagingChannel` abstraction so iMessage/RCS can return later
(`ImessageProviderChannel` is **parked** — do not implement it); SMS segments at ~160 GSM-7 chars so
replies must be terse / link to the full web thread.

Conventions: all messaging code lives under `src/messaging/`. The channel speaks only the
`MessagingChannel` interface so the agent loop never imports Twilio directly. Twilio credentials
(`TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`) come from Wrangler secrets, never code. All inputs/outputs are
Zod-typed. No outbound call throws on a non-2xx Twilio response — return a typed result the caller can log
to the audit stream. Phone numbers are E.164 strings throughout.
