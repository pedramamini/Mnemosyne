# MNEMO-19 — Harness & Tools: Self-authored tools (procedural memory)

Phase 19, closing Track C (see `MNEMO-00-ROADMAP.md`). Goal: procedural memory — the agent persists
reusable scripts to `/brain/tools/`, registers them as callable tools, and re-runs them across sessions.
Depends on MNEMO-16 (tool framework + sandbox-driving registry) and MNEMO-07 (brain filesystem layout +
`git init` in `/brain` + auto-commit helper). Per `docs/PRD.md` v0.5 §6.2 + §4: a self-authored tool is
a script (usually Python) the agent writes once, saves to `/brain/tools/`, and reuses across sessions —
*procedural* memory vs. notes as *declarative* memory. **This is the product's largest security
surface** (agent-authored code re-run later) — contained by complete per-agent sandbox isolation
(§7.3, §8.4); this phase implements that containment carefully.

Conventions: self-authored tools live under `/brain/tools/<name>/` with a manifest declaring name,
description, the entrypoint script, and a Zod-serializable input schema (stored as JSON Schema). On each
turn the registry discovers manifests and dynamically registers them as `ai`-SDK tools whose `execute`
runs the script **inside the agent's own sandbox only** (the isolation boundary from §7.3 — code never
leaves the per-agent container, never touches the Worker/DO process, never reaches another agent's
brain). Authoring a tool auto-commits to the brain git repo (MNEMO-07) so a bad write is restorable
(§6.9). Output obeys the MNEMO-16 large-output-to-path discipline. Authoring emits `tool.authored`;
running emits `tool.ran`.
