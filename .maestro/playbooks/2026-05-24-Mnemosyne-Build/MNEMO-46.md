# MNEMO-46 — Messaging: `onInboundMessage` in the agent DO + async reply + SMS persistence

Phase 46 of Track H (see `MNEMO-00-ROADMAP.md`). Goal: make a text message actually drive the agent — the
per-agent DO receives the normalized inbound message, runs the **same** `streamText`/`generateText` loop as
web chat (same brain, memory, tools — no separate brain), and replies **asynchronously** via the Twilio
REST API once the loop finishes. SMS turns persist to the same store the web UI reads, so text threads
render in-app as first-class conversations with a channel badge. Depends on MNEMO-45 (gateway hands off the
normalized `InboundMessage` + the `onInboundMessage` stub) and MNEMO-15 (the agent loop). Per `docs/PRD.md`
v0.5 §9.3 (reply asynchronously — ack the webhook, run the loop, send via REST; loops exceed webhook
timeouts; use DO `alarm`/`waitUntil`) and §9.5 (per-agent transcript keyed by counterparty; 1:1 bucketed
**one session per calendar day**; each message tagged with `from` + `channel`; web UI reads the same store).

Conventions: the inbound→loop→reply lifecycle lives entirely in `src/agent/` (it IS the agent) but the
send goes back out through a `MessagingChannel` (MNEMO-44) so the DO never imports Twilio directly — the DO
constructs/looks up a `TwilioSmsChannel` for the agent's number. Persistence uses DO SQLite via the
existing `src/agent/sql.ts` schema-init extension point (MNEMO-04). Group-thread handling (multiple agents,
triage gate, coordinator) is OUT of scope here — this phase handles 1:1 SMS only; `threadId === null`.
Access control / capability tiers are MNEMO-47-a — here, assume the sender is allowed (the gateway already
authenticated Twilio).

  **Implementation notes / decisions:**
  - **Deferred via DO alarm, not inline:** `onInboundMessage` persists the inbound + schedules `this.schedule(0, "runInboundReply", …)` and returns fast; the loop runs in the alarm callback (survives hibernation, decoupled from the webhook request lifecycle per §9.3). Running inline would also break the MNEMO-45 gateway test (no model/sandbox mocks → real container).
  - **Reused the web-chat building blocks** (`resolveModel` / `buildSystemPrompt` / `buildTurnTools` / `generateText`) with an `SMS_REPLY_OVERLAY` (terse, link-to-web-thread) and the counterparty's recent transcript as `ModelMessage`s — same brain, no separate loop.
  - **Outbound persistence stores the FULL agent reply** (the web thread is the complete record); the over-SMS body may be truncated + linked by `reply.ts`'s cost guard.
  - **Kept `getLastInboundMessage`/`LAST_INBOUND_KEY`** as a lightweight handoff receipt so the MNEMO-45 gateway test stays green unchanged; the real transcript is additive.
