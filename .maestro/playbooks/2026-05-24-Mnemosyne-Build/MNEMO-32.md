# MNEMO-32 — Frontend foundation: scaffold, design system & shared component library

Phase 32, opening Track G — Frontend (see `MNEMO-00-ROADMAP.md`). This single document builds the entire
**frontend foundation** in one pass: the `frontend/` React + Vite + TypeScript app, the auth-aware API
client, the **single authoritative design language** (design tokens), and the **canonical shared component
library** in `frontend/src/components/ui/` — all *before* any feature screen. Every feature phase (33–43)
then composes these components and never builds bespoke controls. Depends on MNEMO-01 (Worker + Hono
scaffold, so the API the client targets exists). Per `docs/PRD.md` v0.6 §6.5 (conversation-centric,
mobile-responsive parity, agent avatars), **§6.10** (design system & shared components), and §7.4 (Hono +
Zod backend; frontend reuses Crema component patterns).

**Design intent — token-driven so it can be reskinned later:** every component references design tokens for
*every* color, space, font, radius, shadow, and motion value — components NEVER hardcode visual values.
Reskinning later is then a matter of swapping the token file(s), not editing components. Neutral default
skin for now. Each component is presentational, prop-driven, accessible (ARIA + keyboard), strongly typed,
and lives in its own file under `frontend/src/components/ui/` with a co-located `.module.css`.

Conventions: the app lives in `frontend/` (its own `package.json`, separate from the Worker root).
TypeScript strict, ESM, React 18 function components + hooks only. API client in `frontend/src/api/`, shared
UI in `frontend/src/components/ui/`, route-level screens in `frontend/src/pages/`. Keep each module < 500
lines; separate component code / tests / test-runs into their own tasks.
