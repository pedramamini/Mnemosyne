# MNEMO-16 — Harness & Tools: Tool framework & sandbox-driving registry

Phase 16 (see `MNEMO-00-ROADMAP.md`). Goal: the tool framework — a Zod-typed tool registry whose each
`execute` body drives the agent's sandbox (shell / Python / file ops), plus the **large-output-to-FS-
path** pattern that keeps the in-loop message array small. Depends on MNEMO-06 (sandbox provisioning:
per-agent Cloudflare Sandbox, `exec`/`readFile`/`writeFile` wrappers, warm/idle lifecycle, R2
persistence) and MNEMO-15 (the agent loop with its `tools: {}` seam). Per `docs/PRD.md` v0.5 §6.3 (tools
are Zod-typed; each tool's `execute` drives the sandbox via shell/Python/file ops; internal harness
reaches services via a **direct service layer**, no self-API round-trips) + §7.1 (write large tool
outputs to the brain FS and feed back a **path, not the blob** — the SDK does not compact the in-loop
message array, so this is the critical context-discipline rule).

Conventions: tools are built per-turn by `buildTools(env, ctx)` where `ctx` carries the live sandbox
handle + `agentId` + an audit emitter; we use the **direct service layer** trade-off from
`docs/crema-architecture-reference.md` (the "one decision to make deliberately") because Mnemosyne tools
mostly read/write a memory store — no SELF HTTP round-trip per call. Every tool returns a SMALL JSON
result; any output over a threshold is written to `/brain/.tool-out/<id>` and the tool returns the PATH
plus a short head/preview, never the blob. Tool runs emit `tool.ran` audit events (types from
`src/audit/types.ts`).
