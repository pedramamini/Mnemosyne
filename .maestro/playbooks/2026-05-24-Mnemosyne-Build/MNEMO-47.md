# MNEMO-47 — Messaging: Access control, number provisioning & A2P 10DLC

Phase 47 of Track H (see `MNEMO-00-ROADMAP.md`). This document builds the complete **"enable messaging on
an agent"** subsystem in one pass: (1) the **access-control** layer — for every inbound message, whether
the agent accepts it and at what **capability tier** it answers, plus permissive group whitelist
auto-expansion; and (2) **number provisioning** — buying a dedicated Twilio long-code per opted-in agent
and the shared **A2P 10DLC** brand/campaign onboarding US carriers require. Depends on MNEMO-02
(`message_whitelist` + `agent_numbers` D1 tables), MNEMO-44 (`TwilioSmsChannel` Basic-auth pattern), and
MNEMO-45 (the inbound gateway where the accept check wires in + the webhook URL a number points at). Per
`docs/PRD.md` v0.6 §9.1, §9.2, §9.6.

Conventions: access logic in `src/messaging/access.ts` + `src/messaging/tiers.ts`; provisioning in
`src/messaging/provisioning.ts`; 10DLC state in `src/messaging/a2p.ts` + a new migration. The **capability
tier — not the access list — is the real safety boundary** (§9.6). Brand/campaign are **org-level/shared**
(one covers many numbers, §9.2); provisioning is **opt-in per agent** and gated behind the paid add-on
(billing enforcement: MNEMO-49 — guard with a clear extension point here). Twilio calls reuse the Basic-auth
+ `apiBase` pattern from `TwilioSmsChannel` and never throw on non-2xx (typed results). Separate code /
tests / test-runs.
