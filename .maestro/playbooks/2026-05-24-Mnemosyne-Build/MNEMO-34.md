Phase 34 (see `MNEMO-00-ROADMAP.md`). Goal: the home of the app — a list/grid of the user's agents and
the guided **create wizard** that fronts the Discovery flow (a clarify-scope conversation that ends at a
good-enough confidence gate, then provisions the agent). Depends on MNEMO-05 (agent registry CRUD:
`POST /agents`, `GET /agents`, `GET /agents/:id`, `PATCH /agents/:id`), MNEMO-29 (Discovery backend:
clarify-scope conversation + confidence gate + slot rubric), and MNEMO-32 (scaffold). Per `docs/PRD.md`
v0.5 §5 (Discovery → Build → Operation; exit is a soft "good-enough" confidence gate over a rubric of
subject / entity type / sources / cadence / output format), §6.5 (agent avatars; conversation-centric)
and §6.6 (list/filter/search agents).

Conventions: agent-list pages in `frontend/src/pages/agents/`, the wizard in
`frontend/src/components/agents/` + `frontend/src/pages/agents/CreateAgentPage.tsx`, agent API calls in
`frontend/src/api/agents.ts`, discovery API calls in `frontend/src/api/discovery.ts`. The Discovery flow
is a chat-like exchange where the model asks follow-ups until its self-assessed confidence clears the gate;
the UI must surface progress against the soft rubric facets and a clear "create agent" moment when the gate
clears. Separate component code / tests / test-runs.

- [x] Create `frontend/src/api/discovery.ts`: typed client for the MNEMO-29 Discovery backend. Define types for the exchange — `DiscoveryTurn { role: "user" | "assistant"; content: string }`, `DiscoveryState { messages: DiscoveryTurn[]; rubric: { subject: boolean; entityType: boolean; sources: boolean; cadence: boolean; outputFormat: boolean }; confidence: number; ready: boolean }`, and a `DiscoveryResult { agentId: string }`. Implement `startDiscovery(seed: { name: string; description: string })`, `sendDiscoveryMessage(discoveryId, content)` (returns updated `DiscoveryState`), and `finalizeDiscovery(discoveryId)` (called when `ready` — provisions the agent and returns `DiscoveryResult`). Add a clear comment mapping these to the MNEMO-29 endpoint shapes; if MNEMO-29's contract differs, adapt here so the rest of the UI stays stable. Keep this the single adapter point to the Discovery backend.

  _Done, with adaptation._ The real MNEMO-29 contract differs materially and a detailed header comment documents the mapping: Discovery is mounted **per existing agent** (`/agents/:agentId/discovery`), so `startDiscovery` first `POST /agents` to create the draft, then `POST …/discovery/start` — the new agent id becomes the `discoveryId` (returned as `{ discoveryId, state }`). The backend `DiscoveryState` is `{ status, spec, turns }` and `/message` returns `{ reply, state }` (latest assistant reply only, no transcript; rubric/confidence emitted only in the finalized `spec`). So `sendDiscoveryMessage` returns the assistant reply as the new turn(s) + refreshed gate (caller owns the transcript), rubric facets light from `spec.facetNotes`, `confidence` from `spec.confidence` (0 while scoping), `ready = status === "complete"`. There is no literal finalize endpoint — `finalizeDiscovery` maps to `POST …/build` (MNEMO-30 provision, idempotent) and returns `{ agentId }`. This file is the sole Discovery adapter.
