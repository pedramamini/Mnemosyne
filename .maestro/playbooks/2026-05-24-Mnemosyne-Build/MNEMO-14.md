# MNEMO-14 — Harness & Tools: AI Gateway routing, spend caps & secret custody

Phase 14 (see `MNEMO-00-ROADMAP.md`). Goal: route every BYOK provider call through a **Cloudflare AI
Gateway** for request logging, response caching, and **per-user spend caps**, and give BYOK keys real
secret custody (encrypted at rest, never returned to clients). Depends on MNEMO-13 (`getModel()` +
`llm_profiles` + profile routes, which left key custody and Gateway wiring as `TODO(MNEMO-14)`
markers). Per `docs/PRD.md` v0.5 §7.2: AI Gateway for caching, request logs, and per-user spend caps;
**secret custody is a real obligation** (§6.1). Mirrors the Gateway pattern in
`docs/crema-architecture-reference.md` §4 (gateway URL when `AI_GATEWAY_ACCOUNT_ID` is set, direct
fallback otherwise) but adds per-user attribution + caps that Crema lacks.

Conventions: the Gateway is addressed by account id + gateway name from config; provider base URLs are
rewritten to the Gateway path so the SDK providers (OpenRouter/Anthropic/OpenAI) transparently route
through it. Per-user attribution rides on Gateway `cf-aig-metadata` (custom metadata) so logs and caps
key on `accountId`. Spend caps are enforced **before** a call (pre-flight check against accumulated
spend) and updated **after** (from usage). Keys are encrypted with a Worker secret (AES-GCM via
WebCrypto); only the resolver decrypts them, in-process, immediately before constructing the provider.
