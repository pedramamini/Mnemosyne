# MNEMO-23 — Reporting: Code Interpreter integration & chart→PNG pipeline

Phase 23 (see `MNEMO-00-ROADMAP.md`), first phase of Track E (Reporting & Scheduling). Goal: stand up the
`src/reports/` foundation — **persistent Python contexts** in the per-agent sandbox via the Sandbox Code
Interpreter (`createCodeContext` / `runCode`), and a **chart→PNG** rendering pipeline (including SVG→PNG)
that produces a single embeddable artifact. Depends on MNEMO-06 (sandbox provisioning: per-agent Sandbox,
`exec`/`readFile`/`writeFile` wrappers, warm/idle lifecycle, R2 persistence). Per `docs/PRD.md` v0.5 §6.4,
§7.3, §8.1: the Code Interpreter runs persistent Python contexts with rich outputs (charts, tables); charts
are rendered to **PNG** deliberately so one artifact embeds across web, email, and SMS where SVG can't render.

Conventions: all Code Interpreter access goes through one `src/reports/interpreter.ts` wrapper so the rest
of the codebase never calls the (Beta-doc'd) Sandbox SDK methods directly — this isolates the §8.1 caveat
("don't pin business-critical behavior to an undocumented method without a test") to one file. Python
contexts are **per-agent and reused** across a run (matplotlib state, loaded dataframes), keyed by agent so
the warm sandbox amortizes import cost. Outputs are returned as bytes/base64 PNG plus a saved path in the
brain FS — never inline a large blob into the loop message array (§7.1).

- [x] Create `src/reports/charts.ts`: the chart→PNG pipeline. Export `renderChartPng(interp, ctx, spec): Promise<{ pngBytes: Uint8Array; path: string }>` where `spec` is a small Zod-typed chart description (`kind: "line" | "bar" | "scatter" | "pie"`, `title`, `series`/`labels`, optional `xLabel`/`yLabel`). The function builds a matplotlib snippet from `spec`, runs it via `interpreter.runCode`, captures the image/png rich output, decodes base64 to bytes, and writes the PNG to the brain FS under a reports/assets path (e.g. `/brain/reports/assets/<slug>-<ts>.png`) via the MNEMO-06 `writeFile` wrapper, returning both the bytes (for embedding) and the saved path. Keep the Python snippet generation in a pure `buildChartCode(spec)` helper so it is unit-testable without a sandbox.
  - Done. `renderChartPng(interp, ctx, spec, { writer, emitter? })` validates the spec (`ChartSpec.parse`), runs `ensureCharting`, builds + runs `buildChartCode`, finds the `image/png` rich result, decodes base64→`Uint8Array`, ensures the assets dir, persists to `/brain/reports/assets/<slug>-<ts>.png` via `writer.writeFileBytes` (the MNEMO-06 binary write — see the SandboxClient extension note below), and returns `{ pngBytes, path }`. `buildChartCode(spec)` is **pure** (kind-differentiated plotting body; spec embedded as a base64 literal, never interpolated — injection-safe; figure saved Agg-style to a BytesIO then emitted as `image/png` via `IPython.display`, since Agg can't auto-show). New `REPORT_ASSETS_DIR` added to `src/memory/layout.ts` (the single brain-path owner). **Design note (binary write):** `SandboxClient.writeFile` is UTF-8-only and would corrupt PNG bytes, so I added `SandboxClient.writeFileBytes(path, bytes)` (base64 + the SDK's `encoding:"base64"` option) and `renderChartPng` takes a structural `BrainFileWriter` that the client satisfies — keeps the encoded-write in the one FS seam while letting the test inject a spy.
