# MNEMO-05 — Foundation: Agent registry CRUD API

Phase 5 (see `MNEMO-00-ROADMAP.md`). Goal: the authenticated REST surface for managing agents —
create / list / get / update — wired to the D1 registry (MNEMO-02) and the per-agent Durable Object
(MNEMO-04). D1 is the cross-agent source of truth for listing/search; the DO mirrors the operational
subset it needs. Depends on MNEMO-02 (`agents` table + `src/db`), MNEMO-03 (`requireAuth` + current
account id), MNEMO-04 (`MnemosyneAgent` DO + `idFromName` routing). Per `docs/PRD.md` v0.5: one
account → many agents (§6.1); agent has name/description/template/system-prompt/schedule/status (§4,
§6.6); public multi-tenant SaaS, so every route is ownership-scoped to the calling account (§3).

Conventions: all routes mounted under `/agents`, behind `requireAuth`. Request/response bodies are
Zod-validated; agent ids are app-generated `crypto.randomUUID()` (matches MNEMO-02). Every read/write
is filtered by the authenticated `accountId` — a user can only see/touch their own agents (return 404,
not 403, for someone else's agent id to avoid leaking existence). On create/update, write D1 first
(durable registry), then push the operational subset (template, system prompt, schedule) into the DO
via `idFromName(agentId)` so the DO and registry stay consistent.
