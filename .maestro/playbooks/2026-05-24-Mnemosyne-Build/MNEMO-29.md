# MNEMO-29 — Lifecycle: Discovery flow (clarify-scope + confidence gate)

Phase 29 (see `MNEMO-00-ROADMAP.md`). Goal: the **Discovery** stage of the Discovery→Build→Operation
lifecycle. A user gives an agent name + short description; the platform runs a **clarify-scope
conversation** — driven by the same agentic loop as research — that asks follow-ups until it is
confident it understands **what the agent should specialize in**, then persists a structured
**Discovery spec** that MNEMO-30 consumes to provision a live agent. Depends on MNEMO-15 (agent loop:
`streamText`/`generateText` in the `MnemosyneAgent` DO, `stopWhen`, message persistence) and reuses
MNEMO-13's `getModel()` resolver. Per `docs/PRD.md` v0.5 §5(1) and §6.3: exit is a **"good-enough"
confidence gate (~90%)** — the model self-assesses against a **soft rubric of facets** (subject, entity
type, sources, cadence, output format). This is a soft threshold (start when confident), **NOT** a hard
slot-filling form — the model decides when it has enough, and may exit early if the user's first message
is already detailed.

Conventions: Discovery runs *inside* the `MnemosyneAgent` DO (the always-cheap home, no sandbox needed
yet — Discovery is pure conversation + model calls). The confidence gate is implemented as a
**terminator-style tool** (mirrors MNEMO-18's deliberate-exit pattern) whose Zod input schema *is* the
Discovery spec — the model can only "finish Discovery" by emitting a well-formed spec, which both forces
structure and makes the gate verifiable. The spec persists to the DO's `agent_meta` table (MNEMO-04
`setMeta`/`getMeta`) under a `discovery` key, not to D1 (it is DO-resident operating state until Build
promotes the relevant fields to the registry). Keep model prompts in a dedicated module so MNEMO-31's
templates can extend them. All facet scoring is in code/prompt, never a fixed required-fields list.
