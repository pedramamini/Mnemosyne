# MNEMO-20 — Audit Log: wire the spike into an `AuditLog` Durable Object

Phase 20 (see `MNEMO-00-ROADMAP.md`), first phase of Track D ("glass cockpit"). Goal: take the
**existing, tested** audit spike (`src/audit/types.ts`, `src/audit/store.ts`, `test/audit-store.test.ts`)
and run it in the Workers runtime as a dedicated per-agent `AuditLog` Durable Object, plus the ~60-line
SSE fan-out wrapper. Depends on MNEMO-04 (Worker scaffold, `wrangler.toml` binding-stub + `new_sqlite_classes`
migration pattern, `src/env.ts`, vitest workers pool, `idFromName` routing). Per `docs/PRD.md` v0.5 §7.4 and
§8.6: the audit log is a **dedicated `AuditLog` DO per agent** (`idFromName(agentId)`), DO SQLite + FTS5,
fanning out over SSE; the store already runs identically in `node:sqlite` (test) and `ctx.storage.sql` (DO)
behind the `SqlDriver` interface, so this phase is **plumbing the existing logic in**, not re-speccing it.

Conventions: do **not** modify `src/audit/store.ts` or `src/audit/types.ts` logic — they are 6/6 tested.
This phase only adds the DO adapter (a `SqlDriver` over `ctx.storage.sql`), the DO class, the SSE wrapper,
and a binding. One `AuditLog` DO instance per agent via `env.AUDIT.idFromName(agentId)` (separate namespace
from the `AGENT` DO of MNEMO-04 so the audit index is independent and can be queried without waking the loop).
The DO is SQLite-backed (`new_sqlite_classes`) so `ctx.storage.sql` exists.
